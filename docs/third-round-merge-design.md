## 1) 页面合并原则

- 仅在现有模块子功能页承接新需求，避免继续拆新页导致入口分散。
- 统一以“入口页可完成 80% 操作，复杂明细走弹窗/侧滑”。
- 所有列表操作列采用“三层按钮规则”：
  - 一级主动作（高频）`主色按钮`
  - 二级流程动作（审批/确认）`描边状态按钮`
  - 三级管理动作（查看/编辑/删除）`文本链接组`

## 4) 弹窗内容设计（按动作）

## 4.1 采购数据台账-确认入库弹窗

- 顶部：子合同号、合同号、供应商、发货数量
- 中部：实收数量、仓位、入库备注、附件（到货单）
- 底部：确认人（自动）、确认时间（自动）

## 4.2 订单确认推送弹窗

- 字段：订单号、提报单位、场站、指定业务部门、确认人（下拉）、推送说明
- 按钮：`取消` / `确认并推送`
- 提交后：状态改“已确认”，写入确认人/时间

## 4.3 领用审批弹窗

- 字段：申请号、申请部门、物资明细子表、审批意见
- 按钮：`通过` / `不通过`
- 不通过必须填意见

## 4.4 发货出库弹窗

- 字段：发货单号、关联订单、发货数量、收货单位、承运信息、备注
- 校验：发货数量 <= 剩余执行数量
- 提交后：自动扣减剩余执行数量

## 4.5 项目公司验收弹窗

- 字段：验收单号、关联发货、验收数量、验收结果、验收报告上传、验收人
- 通过：更新“已执行数量+N”
- 不通过：生成“重发货任务”

## 5) 审批流统一（BPM最小闭环）

- Flow-A 采购到入库：
  - 订单创建 → 部门确认 → 台账确认入库 → 公司验收确认
- Flow-B 领用到发货：
  - 领用申请 → 公司审批 → 发货出库 → 项目公司验收
- Flow-C 异常回退：
  - 验收不通过 → 重发货 → 再验收

### 5.1 节点角色

- 发起：业务部门/项目公司
- 审批：公司专责（经营发展中心/业务部门负责人）
- 执行：仓储/物流/采购执行人
- 验收：项目公司验收人 + 公司验收人（联合）

### 5.2 状态机建议

- 订单：`待确认 -> 已确认 -> 执行中 -> 已完成`
- 入库：`待确认 -> 已入库`
- 审批：`待审批 -> 已通过/不通过`
- 发货：`待发货 -> 已发货`
- 验收：`待验收 -> 通过/不通过`
